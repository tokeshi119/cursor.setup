# API設計レビュー・チェックリスト

## 目的
- 人間レビューの論点を固定し、雰囲気レビューを防ぐ
- TODO/未確定を可視化して設計判断に回す

## 使うとき
- OpenAPIの叩き台ができた後
- レビュー観点の抜け漏れを防ぎたいとき

## 出力フォーマット（必須）
- 以下テンプレをそのまま使う
- 未確定は TODO を残す

```
### A. 一貫性
- [ ] 同じ概念の型は1つ（ID/日時/Cursor/Moneyなど）
- [ ] 命名が揺れていない（単数/複数、snake/camel、*_id など）
- [ ] エラー形式が全APIで統一されている
- [ ] examples がスキーマと一致している

### B. 画面要件
- [ ] 画面に必要なデータが少ない往復で取得できる
- [ ] 一覧 + サマリ + フィルタが破綻しない
- [ ] ページング/ソート/絞り込みの拡張に耐える

### C. セキュリティ
- [ ] 認証（誰か）と認可（何をしてよいか）が分離されている
- [ ] ロール/スコープが endpoint ごとに説明できる
- [ ] 監査ログ要否が判断されている（管理画面は特に）

### D. 運用
- [ ] エラーが「直し方」まで案内できる
- [ ] rate limit / timeout / retry 方針が説明できる
- [ ] 互換性ポリシー（バージョン/非推奨/移行）がある

### 指摘・懸念
- TODO:
```
