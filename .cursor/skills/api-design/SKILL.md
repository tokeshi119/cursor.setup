# API設計スキル

## 目的
REST API設計を一貫した基準で整理し、設計レビューまで進める。

## 入力
- 対象のユースケース/画面/業務フロー
- 主要リソース（名詞）と関係
- 認証・認可の前提
- 想定クライアント（Web/モバイル/外部連携）
- 非機能要件（レート、キャッシュ、監査、SLA）

## 進め方
1) 目的と利用者を確認する  
2) リソース候補を洗い出し、エンドポイント案を作成する  
3) HTTPメソッド、命名、バージョニング方針を決める  
4) リクエスト/レスポンス、エラー、ページングを定義する  
5) 認証・認可、ロック、レートなど運用面を決める  
6) チェックリストで抜け漏れを確認する  

## 参照
- 詳細ガイドは `references/core.md` を参照する  
- 迷う場合は `references/core.md` のチェックリストに従う  
 - 非同期APIの設計は `references/core.md` の「非同期処理」を参照する  

## 出力フォーマット（詳細）
- 目的/対象範囲
- 主要リソース
- エンドポイント一覧  
  - パス  
  - メソッド  
  - 概要  
- リクエスト仕様  
  - 必須/任意パラメータ  
  - クエリ/ボディ  
  - 例（1件）  
- レスポンス仕様  
  - 成功時ステータス  
  - ボディの例（1件）  
- エラー設計  
  - 代表ステータス  
  - エラーボディの例（1件）  
- ページング方針
- 認証・認可方針
- ロック/レート/キャッシュの方針（必要時）
- 既知の懸念点
- チェックリスト結果

## 例（最小）
「ユーザー一覧と詳細を取得できるAPIを設計する。認証はBearer、ページングが必要。」
