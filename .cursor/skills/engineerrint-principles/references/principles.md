# 実装前プリンシパル詳細

実装前の確認で迷ったときの詳細基準と最小例。

## 責務の分離
- 定義: 1つの変更理由に対して1つの責務を持つ
- チェック:
  - 変更理由が複数あるなら分割する
  - 入力/出力/副作用が同一関心事に閉じているか
- 例:
  - OK: データ取得とUI表示を別モジュールに分離
  - NG: 取得・加工・表示・ログが1関数に混在

## 再利用性
- 定義: 同じ要件を別箇所でも使える形にする
- チェック:
  - 既存の共通部品/ユーティリティで代替できないか
  - 特定画面/機能に過度に結び付けていないか
- 例:
  - OK: 汎用バリデーション関数を共通化
  - NG: 1画面専用の複製ロジックを新規作成

## 保守性
- 定義: 読みやすく、修正しやすい構造
- チェック:
  - 名前と責務が一致しているか
  - 依存が少なく、影響範囲が見えるか
- 例:
  - OK: 依存が明示され、単体で読める
  - NG: 暗黙の依存により挙動が追いづらい

## 変更容易性
- 定義: 変更が局所化され、差し替えが容易
- チェック:
  - 変更点が1〜2箇所に閉じるか
  - 契約（IF/型）が安定しているか
- 例:
  - OK: 抽象IFを通じて実装を差し替え可能
  - NG: 多数の呼び出し元を同時に修正する必要がある

## 車輪の再発明回避
- 定義: 既存の標準・実績ある方法を優先
- チェック:
  - 既存実装/標準ライブラリ/既知パターンの調査を行ったか
  - 新規実装の利点が明確か
- 例:
  - OK: 標準ライブラリ/既存共通の活用
  - NG: 既存で十分な機能を独自実装
